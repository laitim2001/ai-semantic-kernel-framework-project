### 3.0 Architecture (the FAIL==max terminal gains a conditional human-ESCALATE branch; resume gains a verification kind)

```
BEFORE (A1 / 57.98)                                       57.99 (A2 verification-ESCALATE)
  loop._run_turns @ verdict.outcome=="failed_max":         loop._run_turns @ verdict.outcome=="failed_max":
    yield LoopCompleted(verification_failed)  # :2346         if self._verification_escalate_on_max
    return                                                       and not verification_escalated:           # NEW
                                                                  async for ev in _cat10_verification_      # NEW
                                                                      escalate_pause(held=parsed.text,
                                                                      reasons=verdict, ...): yield ev
```

### 3.2 The escalate pause (US-2)
- NEW `async def _cat10_verification_escalate_pause(...)` mirroring `_cat9_output_hitl_pause` (`:1484-1604`): inputs = the held answer text (`parsed.text`), the `_VerifyVerdict` (verifier reasons), the response-snapshot fields (provider/model/tokens — the same set `_replay_approved_output` re-emits), `state`/`ctx`/`session_id`/`messages`/`turn_count`/`verification_attempts`.

  - Create an `ApprovalRequest` (`"kind":"verification"`, `risk_level="HIGH"`, the verifier reason in the payload), emit `ApprovalRequested(HIGH)`, build `pending_approval = {"kind":"verification", "approval_request_id":…, "turn":…, "response_snapshot":…}`, call `_emit_deferred_pause(kind="verification", ..., verification_attempts=…)` which checkpoints + yields `LoopCompleted(AWAITING_APPROVAL)`.

- Integration at the FAIL==max point (`:2344`): `if self._verification_escalate_on_max and not verification_escalated:` → `async for ev in self._cat10_verification_escalate_pause(...): yield ev; return`; `else:` → the A1 terminal (unchanged). Gated additionally on `self._verifier_registry` being present (the gate only runs then anyway).
- DoD: toggle ON + FAIL==max → an `ApprovalRequested(HIGH)` + `awaiting_approval` checkpoint carrying the held answer + reasons; toggle OFF → the A1 terminal byte-identical (regression green).

### 3.3 Resume APPROVE / REJECT-with-note (US-3 / US-4)

The continuation: PASS → deliver; FAIL → terminal `verification_failed`. Exactly one human-coached turn.
- DoD: APPROVE delivers the held answer (a test asserts the delivered content == the held failed answer + the verifier is NOT called on replay); REJECT re-injects the note + runs one turn; reject-then-fail → `verification_failed` (no 2nd `ApprovalRequested`).
