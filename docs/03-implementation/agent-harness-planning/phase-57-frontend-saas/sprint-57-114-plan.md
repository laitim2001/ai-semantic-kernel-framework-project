## 0. Background

Sprint 57.113 opened the Skills System epic (cc-parity `claudedocs/5-status/agent-harness-cc-parity-20260607.md` row 9 — the last systematic CC-parity gap) with the CC-faithful core: a system-bundled `SkillRegistry` (`agent_harness/skills/`, Cat 5) loads `*.md` skills (frontmatter `name`+`description` + a full-instruction body), a cheap "## Available Skills" block rides the `system_prompt` persona seam, and a `read_skill` tool (Cat 2) lazy-loads a skill's full body when the model self-selects. It shipped **system-bundled only** (the block is byte-identical for every tenant; no `tenant_id`, no DB) and explicitly deferred per-tenant catalogs as `AD-Skills-Per-Tenant-Catalog`.

This slice closes that AD: a tenant gets its OWN skills, authored through the admin surface, isolated per tenant, overlaid on the bundled set. It is the FIRST time an `agent_harness` Cat-5 registry is resolved per-tenant from a DB table — a reusable pattern (future per-tenant tool / per-tenant persona catalogs would mirror it) → per `.claude/rules/sprint-workflow.md` §Step 5.5 Day-4 extracts a design note (`32-skills-per-tenant-catalog.md`).

### Design decision (dedicated `tenant_skills` table; a new `platform_layer/skills/` overlay resolver mirroring the config-tiering resolvers; the router swaps the registry source — backend wiring otherwise unchanged; a tenant-settings list-CRUD tab; a multi-tenant drive-through)

- **US-1 is "the overlay primitive"** (Cat 5, pure, no DB): `SkillRegistry` gains `with_overlay(extra: list[Skill]) -> SkillRegistry` — returns a NEW registry containing the base skills with each `extra` skill registered, **a same-name `extra` replacing the base entry** (override-by-name), new names added. Pure function over the in-memory registry (no DB, no I/O) → stays in Cat 5 and keeps the overlay semantics unit-testable in isolation. The bundled `get_default_skill_registry()` singleton is never mutated (the overlay builds a fresh registry each call).
- **US-2 is "the DB-backed catalog"**: a NEW `tenant_skills` table (migration 0030; `tenant_id` + RLS two-policy idiom from invites 0026), a `TenantSkill` ORM (tenant-scoped, mirroring `User`'s `TenantScopedMixin` + UUID PK + timestamps), and a `TenantSkillService` (`platform_layer/skills/service.py`) with table CRUD scoped by `tenant_id` (mirror `TOTPService`: stateless, `AsyncSession` per method, `_set_tenant` before each query). A tenant's skill = `(name, description, instructions)` + a unique `(tenant_id, name)`.
- **US-3 is "the per-request overlay resolver"** (the AD's "resolve like `resolve_session_persona`"): `resolve_tenant_skill_registry(db, tenant_id) -> SkillRegistry` in `platform_layer/skills/service.py` — TTL-cached (the `_ModelPolicyCache` injectable-clock machinery from 57.104), **fail-open to `get_default_skill_registry()`** (db None / error / no rows → the bundled set, byte-identical to today). On a hit: load the tenant's `tenant_skills` rows → `[Skill(name, description, instructions)]` → `get_default_skill_registry().with_overlay(tenant_skills)`. `invalidate_tenant_skill_registry(tenant_id)` (called by every admin mutation) + `reset_skill_registry_cache()` (test isolation, Risk Class C).
- **US-4 is "admin authoring + main-flow wiring"**: (i) **admin CRUD** in `api/v1/admin/tenants.py` — `GET /{tenant_id}/skills` (list), `POST /{tenant_id}/skills` (create), `PUT /{tenant_id}/skills/{skill_id}` (update), `DELETE /{tenant_id}/skills/{skill_id}` (delete); each gated by `require_admin_platform_role`, validates (non-empty kebab `name` / `description` / `instructions`; unique `(tenant_id, name)` → 409/422 on conflict), appends an audit row (`tenant_skill_*`), and `invalidate_tenant_skill_registry(tenant_id)`. (ii) **the router swap** — `api/v1/chat/router.py` awaits `skill_registry = await resolve_tenant_skill_registry(db, current_tenant)` (after the harness-policy resolve) and passes it to `build_handler(...)` instead of `get_default_skill_registry()`. Because `build_handler` already (a) appends `render_catalog_block(skill_registry.list())` to `system_prompt` and (b) passes the registry to `make_default_executor` (which closes `read_skill` over it), **the catalog block + the `read_skill` resolution now carry the tenant overlay with zero change to `handler.py` / `_register_all.py`**.
- **US-5 is "the admin Skills tab"** (full-stack): a tenant-settings "Skills" tab (list-CRUD mirroring `QuotasTab`) — list the tenant's skills, add / edit / delete (name + description + an instructions textarea), against the new admin endpoints via the `tenantSettingsService` client. Admin-internal (exempt from mockup-fidelity per the recon; uses mockup-ui `Card` + `grid-main` + inline-style tokens; English copy).
- **Rejected / deferred**: JSONB-on-`meta_data` storage (the user chose a dedicated table for the DB-backed-catalog scaling path); a rich authoring editor / versioning / hot-reload (`AD-Skills-Authoring-UI`); a per-tenant cache breakpoint for the now-per-tenant catalog block (the block is small + still cache-stable WITHIN a tenant; cross-tenant fragmentation is inherent to per-tenant config and matches model/harness policy — not measured here); a `/skill` slash command (`AD-Skills-Slash-Command`); a dedicated wire event / Inspector "skill loaded" affordance (`AD-Skills-Inspector-Affordance`); tenant SELF-service authoring (this mirrors the config-tiering precedent — `require_admin_platform_role` manages per-tenant config; a tenant-self gate is a later AD); changing `build_handler` / `make_default_executor` / `handler.py` (the router swap is sufficient — lower blast radius).

### 3.1 `SkillRegistry.with_overlay` (US-1)

- **`with_overlay(self, extra: list[Skill]) -> SkillRegistry`** (`registry.py`): build a fresh `SkillRegistry`; register all `self.list()` skills; then for each `extra`, set it by name **replacing** any existing same-name entry (so a tenant skill overrides a bundled one) and adding new names. Implement via a name-keyed dict merge (`{s.name: s for s in self.list()} | {s.name: s for s in extra}`) → register the merged values; return the new registry. Pure, no DB, no mutation of `self` or the bundled singleton. Deterministic order (bundled order then new tenant names, or sorted — Day-0 pins what `list()` ordering the catalog block expects; keep stable).
- Unit-testable in isolation: base ∪ tenant-add → both present; same-name → tenant body wins; empty extra → equals base; the bundled singleton identity is unchanged after an overlay call.

### 3.2 `tenant_skills` table + `TenantSkill` ORM + `TenantSkillService` (US-2)

- **Table `tenant_skills`** (migration 0030; mirror invites 0026): `id UUID PK DEFAULT gen_random_uuid()` · `tenant_id UUID NOT NULL` FK `tenants(id) ON DELETE CASCADE` · `name VARCHAR(128) NOT NULL` (kebab id) · `description VARCHAR(512) NOT NULL` · `instructions TEXT NOT NULL` · `created_at`/`updated_at TIMESTAMPTZ NOT NULL DEFAULT now()` · `UNIQUE (tenant_id, name)` · `INDEX idx_tenant_skills_tenant (tenant_id)`. RLS: `ENABLE` + `FORCE ROW LEVEL SECURITY` + `tenant_isolation_tenant_skills` (USING `tenant_id = current_setting('app.tenant_id', true)::uuid OR … = '{_SYSTEM_SENTINEL}'::uuid`) + `tenant_insert_tenant_skills` (FOR INSERT WITH CHECK `tenant_id = …`). `downgrade` drops the table (RLS/policies fall with it). (Day-0 pins the exact sentinel + policy text.)
- **`TenantSkill` ORM** (`infrastructure/db/models/skill.py` NEW, or `identity.py` per Day-0): `class TenantSkill(Base, TenantScopedMixin)` `__tablename__ = "tenant_skills"`; columns mirroring `User` (UUID PK `server_default=text("gen_random_uuid()")`, timestamps `server_default=func.now()`); `__table_args__ = (UniqueConstraint("tenant_id", "name", name="uq_tenant_skills_tenant_name"), Index("idx_tenant_skills_tenant", "tenant_id"))`. Register in `models/__init__.py` if Alembic metadata requires.
- **`TenantSkillService`** (`platform_layer/skills/service.py`, mirror `TOTPService`): stateless; module singleton `tenant_skill_service`; each method `await _set_tenant(db, str(tenant_id))` first.
  - `async def list(self, db, *, tenant_id) -> list[TenantSkill]` — `select(TenantSkill).where(TenantSkill.tenant_id == tenant_id).order_by(TenantSkill.name)`.
  - `async def create(self, db, *, tenant_id, name, description, instructions) -> TenantSkill` — insert; on the `(tenant_id, name)` unique violation raise a typed `DuplicateSkillError` (mapped to 409/422 at the endpoint).
  - `async def update(self, db, *, tenant_id, skill_id, name?, description?, instructions?) -> TenantSkill` — load scoped by `(id, tenant_id)`; 404 → typed `SkillNotFoundError`; apply provided fields; bump `updated_at`.
  - `async def delete(self, db, *, tenant_id, skill_id) -> None` — delete scoped by `(id, tenant_id)`; missing → `SkillNotFoundError`.

### 3.3 `resolve_tenant_skill_registry` + TTL cache (US-3)

- **`_SkillRegistryCache`** (`service.py`, mirror `_ModelPolicyCache`): `__init__(ttl_s=_DEFAULT_TTL_S, clock=time.monotonic)`; `get`/`put`/`invalidate`/`clear`; `_entries: dict[UUID, tuple[SkillRegistry, float]]`. Module singleton `_cache`.
- **`async def resolve_tenant_skill_registry(db: AsyncSession | None, tenant_id: UUID | None) -> SkillRegistry`**: `if db is None or tenant_id is None: return get_default_skill_registry()`; cache hit → return; else `try:` `rows = await tenant_skill_service.list(db, tenant_id=tenant_id)`; `extra = [Skill(name=r.name, description=r.description, instructions=r.instructions) for r in rows]`; `registry = get_default_skill_registry().with_overlay(extra)`; `except Exception: return get_default_skill_registry()` (fail-open — a DB hiccup never breaks chat); `_cache.put(tenant_id, registry); return registry`. (Empty `extra` → `with_overlay([])` == the bundled set; still cached.)
- **`invalidate_tenant_skill_registry(tenant_id)`** (admin mutations) + **`reset_skill_registry_cache()`** (test isolation; an autouse `conftest` fixture per Risk Class C).
