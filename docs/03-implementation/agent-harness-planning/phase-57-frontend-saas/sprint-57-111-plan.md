### Design decision (trace-aware = stop passing `None`; benchmark = permanent harness over a NEW golden fixture; wire / codegen / DB ALL untouched)

- **US-1 is "thread the real state, then read it"**: the single load-bearing edit is `loop.py:1684` `verifier.verify(output=output_text, state=cast(LoopState, None), trace_context=ctx)` → pass the REAL `state`. The gate `_cat10_verify_gate` (loop.py:1649-1739) is called at :2536-2542 and today only receives `output_text/turn_count/session_id/attempt/ctx` — so US-1 threads the trace (the real `LoopState`, OR a bounded recent-turns + tool-errors view — Day-0 pins which is in scope at the gate caller) into the gate and forwards `state=<real>` to the verifiers. **This is NOT a loop rewrite** — the `_cat10_*` helper family already exists (A1); US-1 changes one argument + the helper signature + the judge's prompt builder.
- **The candidate answer is NOT yet in `state.transient.messages` at gate time** (it is appended at :2552-2555, AFTER the gate call at :2536-2542) → the judge gets `output` (the candidate) + `trace` (the PRIOR turns + tool errors) cleanly, no double-counting (Day-0 confirms the append order).
- **The judge consumes the trace; the rules verifier does not**: `LLMJudgeVerifier._build_prompt(output)` (`llm_judge.py:112-120`) does `template_text.replace("{output}", output)`. US-1 extends it to ALSO build a bounded `{trace}` block from `state.transient.messages` (recent N) + tool-error results and substitute it. When `state is None` (a non-loop call / back-compat), `{trace}` resolves empty → graceful no-op. `RulesBasedVerifier` (`rules_based.py:51-73`) keeps ignoring `state` by design.
- **Trace is bounded** (token discipline — the judge runs on cheap tier but a runaway trace inflates cost): a recent-turns window (default ~6 messages / 3 turns) + the tool-error results within it, capped by a char/token budget. A module constant with an env override (NOT a per-tenant policy — that is C3 territory, out of scope).
- **US-2 benchmark = permanent eval harness** (user decision 2026-06-13): a NEW labeled golden fixture `tests/fixtures/verification/judge_benchmark.yaml` (~24-32 cases, mirroring the `tests/fixtures/guardrails/*.yaml` precedent) with `category ∈ {clear_pass, clear_fail, trace_dependent, borderline}`; the `trace_dependent` cases carry a `trace` (recent turns + a tool error) whose FINAL string looks fine but is wrong given the trace — these exercise US-1 AND let the benchmark report a **trace-aware-vs-final-only delta** (the coupling that proves trace-awareness helps). A `@pytest.mark.benchmark` real-LLM test (`tests/benchmark/test_judge_accuracy.py`) builds a real Azure `ModelProfile`, runs each case through `profile.cheap` + `profile.action` judges, computes cheap-accuracy / strong-accuracy / cheap-vs-strong-agreement / per-category breakdown + the trace delta, asserts cheap-accuracy ≥ a tracked threshold, and writes a JSON+markdown report. The `benchmark` marker is registered + CI-deselected (real-LLM, token-cost, on-demand).
- **Rejected**: a new wire field surfacing WHICH turn/tool-error the judge flagged (keep the wire stable — `reason` carries it as prose; proposal §1.3); making RulesBasedVerifier trace-aware (it is rule-based — string is its domain); a per-tenant trace window / per-tenant judge template (C3 — out of scope); a labeled-data column on `verification_log` (the benchmark uses a separate curated golden fixture, NOT production logs — production logs have no ground-truth label); an automatic threshold-driven judge-tier auto-switch (the design-note-24 decision is a HUMAN go/no-go on the measured number, not a runtime switch); a one-off throwaway benchmark (user picked permanent).

### 3.1 Trace-aware critique verifier (US-1)

- **Thread the state**: `_cat10_verify_gate` gains the trace (Day-0 pins: the real `LoopState` if in scope at `:2536`, else a bounded `(messages, tool_errors)` view assembled at the call site); the gate forwards it to `verifier.verify(state=<real>)` (replacing `cast(LoopState, None)` @ `:1684`). `loop.py` diff = one call-site arg + one helper-signature param (NOT a rewrite; the `_cat10_*` family + `_VerifyVerdict` stay).
- **Trace builder** (`verification/_trace.py` NEW): `build_trace_block(messages, *, max_turns=DEFAULT, char_budget=DEFAULT) -> str` — formats the last `max_turns` worth of `messages` (role + concise content) and highlights tool-error results, capped by `char_budget`. Provider-neutral (operates on `Message` list — no SDK import). Constants with env overrides (`CHAT_VERIFICATION_TRACE_TURNS` / `_TRACE_CHAR_BUDGET`); NOT per-tenant.
- **Judge prompt**: `LLMJudgeVerifier._build_prompt(output, state)` builds the trace via `build_trace_block(state.transient.messages, ...)` when `state` is non-None, substitutes `{trace}` in the template; `state is None` → `{trace}` empty. `output_quality.txt` gains a `{trace}` section instructing the judge to weigh the trace (tool errors / prior turns) alongside `{output}`. Back-compat: a template WITHOUT `{trace}` still works (`.replace` no-op).
- **RulesBasedVerifier**: UNCHANGED (string-only by design).
- **Wire**: UNCHANGED — `VerificationFailed.reason`/`suggested_correction` carry the trace-aware critique prose; no new event/field/codegen. (Recon: the wire is stable; an optional "which-turn" field is explicitly rejected.)
- **Tests**: judge sees a tool error in the trace → critiques it (the reason references the trace) · `state=None` → empty trace, judge still runs (back-compat) · trace bounded to N turns (a long history is truncated) · the gate forwards the real state (the `cast(None)` is gone — a gate-level test) · existing `test_loop_verification_gate.py` / `test_llm_judge.py` convert if any asserts `state is None` (Never-Delete → convert).

### 3.2 Cheap-judge accuracy benchmark — permanent harness (US-2)

- **Golden fixture** (`tests/fixtures/verification/judge_benchmark.yaml` NEW, ~24-32 cases): each `{id, output, trace?(list of {role, content}), expected_passed: bool, category}`. Categories: `clear_pass` (good answer), `clear_fail` (wrong/empty/format-broken answer), `trace_dependent` (final string looks fine but a tool error / prior-turn contradiction in `trace` makes it wrong — these REQUIRE trace-awareness to judge correctly), `borderline` (genuinely ambiguous — measures over/under-correction). Mirror the `tests/fixtures/guardrails/*.yaml` shape; labeled by hand with care (parent-direct — labeling is judgment).
- **Harness** (`tests/benchmark/test_judge_accuracy.py` NEW, `@pytest.mark.benchmark`): builds a real Azure `ModelProfile` (`build_azure_model_profile`); for each case runs a `profile.cheap` judge and a `profile.action` judge (trace-aware — passes the case `trace` as a `LoopState`-equivalent so trace-dependent cases exercise US-1); computes **cheap_accuracy** (cheap vs label), **strong_accuracy** (strong vs label), **cheap_vs_strong_agreement**, **per-category accuracy**, and a **trace_delta** (trace-aware vs final-only accuracy on `trace_dependent` cases — proves US-1's value); accumulates judge tokens (from `VerificationResult.input_tokens/output_tokens`) for a cost line; asserts `cheap_accuracy >= BENCHMARK_CHEAP_ACCURACY_FLOOR` (a tracked constant — the design-note threshold A3 defines); writes a JSON + markdown report to `tests/benchmark/reports/` (gitignored) AND returns a summary for the design-note verdict.
- **Marker + CI**: register `benchmark` in `pyproject.toml [tool.pytest.ini_options].markers` (or `pytest.ini` — Day-0 pins); CI's pytest invocation deselects it (`-m "not benchmark"` — Day-0 confirms / adds). Runs on-demand: `pytest -m benchmark` with real Azure env.
- **Re-runnability**: the harness is config-free (reads the fixture + the env model profile) → re-run on any judge/model/template change to re-measure. Optional thin `backend/scripts/benchmark_judge.py` wrapper that calls the same core (decided Day-1 — only if it adds value beyond `pytest -m benchmark`).
- **Tests of the harness itself** (CI-safe, NOT benchmark-marked): the fixture loads + schema-validates (every case has the required keys + a valid `category`); the metric computation is pure-function-tested with a MockChatClient (cheap/strong agreement math, trace_delta math) so the harness logic is covered WITHOUT real LLM.

## 8. Dependencies & Risks

| Risk | Mitigation |
|------|------------|
| The full `LoopState` is NOT cleanly in scope at the gate caller `:2536` → threading it is invasive | Day-0 Prong-2 reads `_run_turns` around `:2536`; if `state` is not directly available, assemble a bounded `(messages, tool_errors)` view at the call site and pass THAT (the ABC's `state` param can wrap it, or add a typed trace param to the gate helper — still NOT a loop rewrite) |
| Threading the candidate answer into the trace double-counts it (judge sees the answer twice) | Day-0 confirms the candidate is appended AFTER the gate (`:2552-2555`) → `state.transient.messages` at gate time excludes it; the trace builder only reads PRIOR turns |
| Trace inflates judge token cost (cheap tier but unbounded history) | `build_trace_block` caps at `max_turns` + `char_budget`; a unit test pins truncation; the benchmark reports the token cost so a regression is visible |
| `output_quality.txt` `{trace}` section changes the judge's pass/fail behavior on EXISTING cases (a regression in normal verification) | the benchmark's `clear_pass`/`clear_fail` categories measure exactly this; the back-compat path (`state=None` → empty `{trace}`) keeps non-loop calls identical; existing smoke tests assert continued function |
| A `state is None` assertion in an existing test breaks when state becomes real | Day-0 greps the verification test suite for `state` assertions; convert (Never-Delete), do not skip |
| `benchmark` marker not deselected in CI → real-LLM benchmark runs in CI (no Azure creds → fail, or burns tokens) | Day-0 pins the CI pytest command; register the marker + ensure `-m "not benchmark"`; a CI-safe harness-math test (MockChatClient) covers the logic so CI still validates the harness |
| The golden labels are subjective (borderline cases) → an unstable accuracy number | label conservatively; keep `borderline` a SEPARATE category (reported, not in the pass/fail floor); the floor is asserted on `clear_*` + `trace_dependent` where the ground truth is unambiguous |
| No numeric threshold exists (design note 24 is qualitative) → an arbitrary floor | A3 DEFINES the floor as a tracked constant with a documented rationale; the first real run calibrates it; the verdict is a human go/no-go, not a hard gate |
| Real-LLM benchmark is non-deterministic (judge temperature) → flaky accuracy | run the judge at temperature 0 if the seam allows (Day-0 checks the ChatRequest); report the run is a snapshot, not a CI gate; re-runnable for re-measurement |
| dt leg A: forcing a deterministic trace-dependent fail | reuse the "soft forced-fail judge template" (carryover :140) + "forced-fail steers no-tools" (:155) lessons; construct a prompt that produces a tool error then a superficially-fine answer; if non-deterministic, the `forced_fail_trace.txt` strict template makes the judge cite the trace deterministically |

| `loop.py` diff creeps beyond a threading change into a logic edit | line-by-line `git diff` review at the gate; if the trace needs more than an arg, STOP and re-scope (keep A3 a threading + judge-prompt slice, not a loop refactor) |
