  問題 3 討論：為什麼自建路由而非用 MAF

  你的直覺是對的。自建路由有很好的架構理由：

  PoC Orchestrator 的核心設計是「混合式」：
  Step 1-3: CODE-ENFORCED（確定性，一定要跑）
  Step 4:   LLM-DRIVEN（只有路由決策用 LLM）
  Step 5-6: CODE-ENFORCED（確定性，一定要跑）

  MAF HandoffBuilder 的設計是「全 LLM 驅動」：
  LLM 決定 → 要不要讀 memory → LLM 決定 → 要不要搜知識 → LLM 決定 → 路由
  （每一步都由 LLM 判斷，沒有強制執行的概念）

  差異的關鍵：

  ┌──────────┬───────────────────────────────┬───────────────────────────────┐
  │   特性   │           PoC 自建            │      MAF HandoffBuilder       │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ 強制步驟 │ ✅ Step 1,2,3,5,6 必定執行    │ ❌ 所有步驟由 LLM 決定        │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ 可預測性 │ ✅ 6 步固定流程               │ ❌ LLM 可能跳過步驟           │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ 成本控制 │ ✅ 只 1 次 LLM call（Step 4） │ ❌ 每次 handoff 都是 LLM call │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ 延遲     │ ✅ ~20s（1 次 LLM）           │ ❌ 多次 LLM = 更慢            │
  ├──────────┼───────────────────────────────┼───────────────────────────────┤
  │ 可觀測性 │ ✅ 每步都有明確 log           │ ❌ LLM 內部決策不透明         │
  └──────────┴───────────────────────────────┴───────────────────────────────┘

  企業場景的需求：在 IT Ops 環境中，memory 讀取、知識搜尋、意圖分析是強制性的 SOP，不能讓 LLM
  決定跳過。自建路由恰好滿足這個需求。

  所以 PoC 的架構是合理的 — 它不是「不知道 MAF 有路由」，而是刻意選擇了「確定性步驟 + LLM 路由」的混合模式。MAF 的
  HandoffBuilder 更適合純 LLM 對話場景（如客服 chatbot），但企業 orchestrator 需要更強的控制力。

  你覺得這個分析對嗎？或者你有其他想法？
