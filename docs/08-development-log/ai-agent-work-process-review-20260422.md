  流程順序分析

  圖中規範順序：感知 → [記憶⇄規劃⇄工具] → 行動 → 響應（回饋）

  本項目實際順序（Phase 48 LLM-native orchestrator）：
  Request → Intent Router → Dispatch → [MAF/SDK/Team] → Tools/Memory → Action → SSE
                                            ↑ (mem0 查詢在 agent 內部觸發)

  主要差距

  1. 無獨立「感知層」 — 輸入直接進 intent router，缺統一 preprocessing/normalization
  2. 無「自我檢查」機制 — Agent 不會主動 reflect 結果並修正（CC 的 <thinking> 模式缺席）
  3. 工具覆蓋不全 — 圖像生成缺、網絡瀏覽有限
  4. 閉環未閉合 — 行動 → 感知 的回饋路徑在圖中是虛線，本項目靠 SSE 單向推送，缺 agent-internal 的 outcome evaluation

  建議優先級

  1. 🟡 Phase 49 候選：加入 Perception 預處理層（normalize user_req + env_state + last_response）
  2. 🔴 高優先：引入 Self-Check / Reflection loop（配合 Phase 47 執行日誌已具備基礎）
  3. 🟢 低優先：補齊圖像生成與網絡瀏覽工具

  結論

  本項目架構元素覆蓋約 75%（10/14），核心 Agent/Memory/Planning/Tools/Action 完整；流程順序大致正確但缺 Perception 前置與 Self-Check 後置回饋，這與     
  Phase 48 LLM-native orchestrator 方向一致，建議 Phase 49 聚焦閉環補齊。

  關鍵誤會點

  1. Pipeline 的 8 步不等於 Agent 的思考循環
  Pipeline 是**「進 Agent 之前的守門員」**（memory preload + intent 分類 + 風險 + HITL）。真正的 agent 思考發生在 Step 7 內部。
  2. 「記憶讀取」是 preload，不是 agent ⇄ memory
  圖中是雙向箭頭（agent 隨時存取），本項目是單向注入（開場灌好）。agent 執行中沒有 memory.query() 工具。
  3. 「LLM 選路」≠ 規劃
  Step6 只是三選一分類器（direct_answer / subagent / team），不分解任務、不生成 plan、不列 subgoals。
  4. 閉環完全不存在
  行動 → 響應 後沒有回到感知。Pipeline 跑完就結束，不會因結果不好觸發第二輪。

  }

  建議架構：Agentic Router + 三路 RAG

  用戶問題
      ↓
  Query Classifier (LLM)
      ├── 分類為「單點事實」→ Traditional RAG（快、便宜）
      ├── 分類為「關聯分析」→ Graph RAG（GraphRAG / Neo4j）
      ├── 分類為「複雜推理」→ Agentic RAG（多輪檢索 + 驗證）
      └── 分類為「混合」    → 並行三路 + LLM 合成
      ↓
  回傳結果給 Agent Loop

  對本項目的具體建議

  1. 保留你現有的向量 RAG（作為 Layer 1 fast path）
  2. 新增 Graph 層 — 你已經有 graphify，可以：
    - 企業資料抽實體（客戶、合約、產品、人員）
    - 建關係圖（Neo4j 或 graphify 底層）
    - 用 Microsoft GraphRAG 的社群摘要模式做 global queries
  3. Agentic 層 = 你的 agent loop 本身
    - 不需要獨立「Agentic RAG 模組」
    - Agent loop 可以呼叫 vector_search + graph_query + keyword_search 工具
    - LLM 自主決定用哪個、怎麼組合
  4. Query Classifier 作為前置
    - 簡單查詢不進 agent loop，直接查回
    - 複雜查詢才進 loop，這樣省成本

  你的 KB enterprise worktree 應該產出什麼

  memory 中 project_kb_enterprise_worktree.md 指向 research/knowledge-base-enterprise 分支。該 worktree 最終應該決策：

  - 採用哪個 Graph 框架（Neo4j / ArangoDB / graphify 自擴）
  - 實體抽取策略（預先批次 vs 動態）
  - 三路 RAG 的路由規則（哪類問題走哪路）
  - Permission-aware retrieval（按 role 過濾）
  - Freshness（文件更新如何同步到 graph）
  - 成本模型（embed / graph build / query 各多少）

  這個 worktree 的產出會直接餵給 Layer 1 合約（ToolSpec 中會定義 knowledge_search / graph_query / entity_expand 工具）。
