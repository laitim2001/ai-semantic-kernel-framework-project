## 一、執行摘要

### 1.1 核心判斷

IPA Platform 是一個**架構設計極具前瞻性**的企業 AI Agent 平台原型。其 11 層架構、三層路由成本優化、MAF+Claude SDK 混合編排、MCP 統一工具層、AG-UI 即時通訊協議等設計理念，在企業 Agent 平台領域中處於領先水平。然而，從企業落地角度而言，平台目前處於**「技術展示可行、內部試用尚需加固」**的階段。

### 1.2 業務結論

| 維度 | 結論 |
|------|------|
| **業務價值** | 高 — RAPO 的 IT 運維痛點（重複性工單、多系統切換、事件關聯分析）與 IPA Platform 的核心設計高度契合 |
| **技術可行性** | 中 — 架構骨架完備，但安全加固、持久化替換、端口修復等基礎工程仍需 2-3 週 |
| **落地時機** | 建議 Phase A（安全基礎）完成後即開始內部小範圍試用，預計 3 週後 |
| **投資回報** | 正向 — 以 RAPO 團隊規模和使用頻率估算，6 個月內可達投資平衡點 |
| **最大風險** | IT 安全審查 — 7% Auth 覆蓋率若在審查前暴露，可能影響項目推進 |

### 1.3 一句話建議

> **先修路、再展示、漸進落地**：用 3 週修復安全基礎（Phase A），選擇「AD 帳號管理 + LDAP MCP」作為第一個完整場景進行內部展示，再逐步擴展到 ServiceNow RITM 處理和跨系統事件分析。

## 二、RAPO 場景重新評估與優先級排序

### 2.1 排序方法論

原報告按「場景完整度」排序。本報告從業務分析角度，引入四維評估框架進行重新排序：

| 評估維度 | 權重 | 說明 |
|----------|------|------|
| **業務價值** | 35% | 對 RAPO 日常運營的直接影響：時間節省、錯誤減少、可見度提升 |
| **技術可行性** | 25% | 現有技術基礎的就緒程度：MCP 是否存在、Mock 比例、端到端流程完整性 |

| **風險可控性** | 20% | 出錯的影響範圍：是否涉及生產環境、客戶數據、合規敏感操作 |

#### 場景一（新）：AD 帳號管理與 RITM 自動處理

**業務需求**：
RAPO IT 團隊每天處理大量 AD 帳號相關的 ServiceNow RITM（Request Item），包括：新員工入職帳號建立、群組權限變更、離職帳號停用、密碼重設。這些是**高頻率、低風險、高度結構化**的任務，非常適合作為 AI Agent 自動化的第一個落地場景。

**為何排在第一**：
1. **LDAP MCP 已存在且可用** — 5 個檔案的完整 AD 查詢/修改能力，無需從零開發
2. **ServiceNow Webhook Handler 已存在** — 雖有 Mock 版本，但真實版的骨架完整
3. **MAF HandoffBuilder 是平台最成熟的模組** — 23 個 Builder、8 個有真實 MAF import
4. **風險可控** — AD 群組成員變更是低風險操作，即使 Agent 出錯也容易回滾
5. **業務價值立即可見** — 每個 RITM 處理從平均 30 分鐘縮減到 3-5 分鐘
6. **不需要 ServiceNow MCP** 即可啟動 — 第一版可以只處理「接收 RITM → 執行 AD 操作」，RITM 狀態更新可手動或透過 n8n 完成

**技術可行性**：

| 步驟 | 可行性 | 說明 |
|------|--------|------|
| ServiceNow Webhook 接收 | ✅ | Handler 存在 |
| RITM 內容解析 | ✅ | 結構化 JSON，PatternMatcher 可處理 |
| 風險評估 | ✅ | RiskAssessor 真實運行，AD 變更為 LOW/MEDIUM |
| Agent 執行 AD 操作 | ✅ | LDAP MCP 存在 |
| 結果驗證 | ✅ | LDAP 查詢驗證操作結果 |
| RITM 狀態回寫 | ⚠️ | 需 ServiceNow MCP 或 n8n 中轉 |

**風險評估**：

| 風險項 | 等級 | 緩解策略 |
|--------|------|---------|
| AD 操作錯誤 | 低 | HITL 審批 + AD 操作的原子性回滾 |
| 權限過度授予 | 中 | LDAP MCP 的 permission patterns + 白名單群組 |
| 帳號數據洩漏 | 中 | LDAP 查詢結果脫敏 + Auth 加固 |

#### 場景二（原場景一）：IT 事件處理與自動診斷

**業務需求**：
RAPO 管理多個 Azure 環境，ETL Pipeline（如 APAC Glider）的失敗、VM 異常、服務中斷等事件需要快速診斷和處理。目前依賴人工排查 Azure Portal + 日誌分析，耗時 2-4 小時。

**為何降為第二**：
1. 原報告的場景一需要 Azure 憑證配置、API Key、CORS 修復等多項前置工作
2. Azure MCP 調用的權限管理更複雜（涉及多個 subscription）
3. 涉及**生產環境操作**，風險較高
4. 需要 Claude Extended Thinking（Anthropic API Key）才能發揮「自動診斷」價值

**技術可行性**：

| 步驟 | 可行性 | 說明 |
|------|--------|------|
| PatternMatcher 識別事件類型 | ✅ | 需新增 ETL/Azure 事件規則 |
| RiskAssessor 評估 | ✅ | 生產環境事件會被評為 HIGH/CRITICAL |
| HITL Teams 通知 | ⚠️ | 需 Teams Webhook 配置 |
| Azure MCP 查日誌 | ⚠️ | 需 Azure 憑證和 RBAC |
| Shell/SSH 查下游狀態 | ⚠️ | 需認證保護 |
| Claude 診斷分析 | ⚠️ | 需 Anthropic API Key |
| AG-UI SSE 串流 | ❌→⚠️ | Phase A 修復 CORS 後可用 |

**風險評估**：

| 風險項 | 等級 | 緩解策略 |
|--------|------|---------|
| 生產環境誤操作 | 高 | HITL 強制審批 + 只讀模式優先 |
| Azure 權限過大 | 高 | 最小權限原則 + Service Principal |
| 診斷結果誤導 | 中 | Claude 建議需人工確認，非自動執行 |

#### 場景三（原場景二）：Swarm 多 Agent 協作分析

**業務需求**：
安全基礎設施審計、合規檢查、多維度系統分析等需要多個專家角色協同工作的場景。

**為何排在第三**：
1. Swarm 是**展示效果最佳**的場景 — 多 Worker 即時進度、前端 Panel 16 個組件、9 種事件串流
2. 但 Swarm **不在主流程中**，需獨立 Demo API 觸發
3. 代碼品質最高（Thread-safe、4 類測試），整合到主流程是確定性的工程工作
4. 適合作為 Phase B 完成後的「進階展示」功能

**技術可行性**：

| 步驟 | 可行性 | 說明 |
|------|--------|------|
| Swarm 觸發 | ✅ | Demo API 可用 |
| Worker 並行執行 | ✅ | 但使用「模擬進度」非真實 Agent |
| SSE 串流到前端 | ✅ | Swarm 事件透過 CustomEvent 橋接 |
| 前端 Panel 展示 | ✅ | 16 個組件、品質最高 |
| 真實 Agent 執行 | ❌ | 需整合 ClaudeCoordinator |

#### 場景四（原場景四）：跨系統事件關聯分析

**業務需求**：
系統異常時關聯 Azure Monitor + ServiceNow + 歷史案例，找出根本原因。這是 RAPO 業務價值最高的場景（可減少 MTTR 50%+），但技術上完全不可行。

**為何排在最後**：
1. Correlation Analyzer 的 5 個數據方法**全部返回假數據**
2. Root Cause Analyzer **硬編碼 2 個歷史案例**
3. 即使上層分析邏輯是真實的，沒有真實數據輸入 = 輸出完全不可信
4. 修復路徑最長：需連接 Azure Monitor API + ServiceNow CMDB + 建立歷史案例庫

### 3.3 成功標準

| 指標 | 目標 | 測量方式 |
|------|------|---------|
| RITM 處理時間 | 從 30min 降到 5min（自動）或 10min（需審批） | 計時對比 |

1. **第一版不含 RITM 狀態回寫**：ServiceNow MCP 尚未開發，RITM 狀態更新暫由 n8n 工作流或人工處理

## 四、業務價值量化分析

### 4.1 各場景業務價值量化表

| **日處理量** | 15-20 個 AD RITM | 同量但自動化 | - |

## 六、n8n 協作策略

### 6.1 現狀分析

RAPO 已有運行中的 n8n 實例，處理定時觸發、Webhook 接收、簡單工作流自動化。PRD 中明確定義了 n8n 作為「事件驅動入口層」的角色（Feature 8: n8n 觸發與錯誤處理，P0 優先級）。

### 6.2 IPA + n8n 分工定義

**核心原則：n8n 做「觸發和編排」，IPA 做「推理和執行」**

| 能力 | n8n 負責 | IPA Platform 負責 |
|------|---------|-------------------|
| **定時觸發** | Cron 表達式、排程管理 | 接收觸發事件，執行 Agent 邏輯 |
| **事件接收** | Webhook 入口、事件過濾 | 意圖識別、風險評估 |
| **簡單自動化** | 固定流程（IF/THEN/ELSE） | 需要 AI 推理的複雜決策 |
| **錯誤處理** | 重試策略、DLQ 管理 | Agent 執行錯誤的智能恢復 |
| **系統串接** | HTTP 節點、API 調用 | MCP 統一工具層（安全隔離） |
| **狀態回寫** | ServiceNow API 調用 | Agent 執行結果生成 |

### 6.3 協作模式設計

```
模式 A：n8n 觸發 → IPA 執行
──────────────────────────────
ServiceNow RITM 提交
    → n8n Webhook 接收
    → n8n 過濾（只處理 AD 變更類型）
    → n8n HTTP 節點調用 IPA API
    → IPA 執行 Agent 邏輯
    → IPA 返回結果
    → n8n 更新 ServiceNow RITM 狀態

適用：所有外部系統觸發的自動化

模式 B：IPA 執行 → n8n 後處理
──────────────────────────────
用戶在 IPA Chat 提交請求
    → IPA 執行 Agent 邏輯（AI 推理 + MCP 操作）
    → IPA 調用 n8n Webhook 觸發後處理
    → n8n 執行固定流程（發郵件、更新報表、通知相關人員）
    → n8n 返回完成狀態

適用：Agent 執行後需要多步驟後處理的場景

模式 C：n8n 定時 + IPA 巡檢
──────────────────────────────
n8n Cron 每小時觸發
    → 調用 IPA Patrol Agent
    → IPA 執行 5 種巡檢（健康、性能、安全、合規、成本）
    → 如發現異常 → IPA 觸發事件處理流程
    → IPA 通知結果
    → n8n 記錄巡檢歷史到 DB

適用：定時監控和預防性維護
```

### 6.4 分階段整合路線

| 階段 | 整合方式 | 說明 |
|------|---------|------|
| **Phase A** | 無整合 | n8n 繼續獨立運行，IPA 專注安全加固 |
| **Phase B** | 模式 A 基礎版 | n8n 透過 HTTP 節點調用 IPA API，IPA 處理 AD RITM |
| **Phase C** | 模式 A+B 完整版 | 雙向整合，n8n MCP 開發完成，IPA 可主動觸發 n8n |
| **Phase D** | 三種模式全部啟用 | 包含定時巡檢、後處理、雙向觸發 |

### 6.5 避免的陷阱

1. **不要讓 IPA 取代 n8n**：n8n 在簡單自動化場景中已經足夠好用，且 RAPO 團隊已熟悉。IPA 的價值在於「n8n 做不到的 AI 推理」
2. **不要重複建設觸發機制**：n8n 的 Cron/Webhook 已經穩定運行，IPA 不需要自建觸發層
3. **避免環形依賴**：IPA 調用 n8n → n8n 調用 IPA → IPA 調用 n8n... 需要明確的流程終止條件
4. **統一監控**：IPA 和 n8n 的執行日誌需要可在同一個 Dashboard 查看

   - HITL 審批機制（高風險操作需人工確認）
   - Rate Limiting 和 CORS 的配置
4. 限定審查範圍為「AD 帳號管理場景」，不提交全平台
5. 準備「持續改進計劃」，展示 Phase C 的進一步安全加固路線

**應急方案**：如果仍被否決，可退回到「純 Demo 環境」模式——在隔離的測試網路中運行，不連接生產 AD。

#### R6: LDAP Service Account 權限審批延遲

**緩解行動**：
1. Phase A 第 1 週即提交 Service Account 申請
2. 申請時明確說明：
   - 最小權限原則：只需 AD 群組成員的讀取/寫入權限
   - 不需要：創建/刪除 OU、修改 GPO、重設密碼（初期）
   - 監控：所有操作通過 IPA 審計追蹤
3. 先申請測試 AD 環境的 Service Account（通常審批較快），用於開發和 Demo
4. 平行進行生產環境的 Service Account 申請

#### R7: 單人/小團隊維護 276K LOC 的壓力

**緩解行動**：
1. **專注策略**：不維護全部 276K LOC。只維護與當前場景相關的模組：
   - 核心：orchestration/ + hybrid/ + mcp/ldap + ag_ui/（~60K LOC）
   - 場景：ServiceNow Handler + PatternMatcher + RiskAssessor（~5K LOC）
   - 基礎：auth + infrastructure/（~10K LOC）
2. **Claude Code 持續輔助**：利用 Claude Code 的代碼生成能力進行維護
3. **自動化測試**：為核心路徑建立 E2E 測試，確保每次修改不破壞現有功能
4. **技術債管理**：建立技術債 backlog，每個 Phase 分配 20% 時間處理技術債
5. **文件優先**：V7 分析報告和本報告作為知識庫，降低未來 onboarding 成本

## 十一、報告可能遺漏的業務考量

### 11.1 原報告未充分涵蓋的面向

#### 11.1.1 跨時區運維場景

RAPO 管理的是 Asia Pacific 區域的 IT 系統，涉及多個時區（日本、澳洲、東南亞）。原報告未考慮：
- HITL 審批的時區問題：30 分鐘超時後直接 EXPIRED，如果審批者在不同時區（例如深夜），審批可能永遠無法完成
- **建議**：HITL 超時應根據審批者所在時區和工作時間動態調整；超時後應升級到次級審批者，而非直接 EXPIRED
- **影響**：在 Phase B 的 HITL 增強中需要加入時區感知邏輯

#### 11.1.2 語言處理能力

RAPO 的用戶可能使用英文、日文、中文提交請求。原報告提到 PatternMatcher 的「多語言場景支持」問題，但未深入：
- 目前 PatternMatcher 的關鍵詞匹配主要針對英文
- Claude SDK 天然支持多語言，但 PatternMatcher 和 SemanticRouter 需要擴展
- **建議**：Phase B 中為 PatternMatcher 增加多語言規則庫；Phase C 啟用 SemanticRouter 真實版（支持多語言 embeddings）

#### 11.1.3 與現有 ITSM 流程的衝突

RAPO 已有成熟的 ITSM 流程（基於 ITIL）。引入 AI Agent 可能與現有流程衝突：
- Change Management：Agent 對 AD 的操作是否需要走 Change Request 流程？
- Incident Management：Agent 自動診斷是否需要在 ServiceNow 中建立對應的 Incident 記錄？
- Problem Management：Agent 的 Root Cause 分析結果是否需要與現有 Problem 記錄關聯？
- **建議**：在 Phase B 設計中，確保 Agent 的每個操作都在 ServiceNow 中有對應記錄，不繞過現有 ITSM 流程

#### 11.1.4 數據主權和數據駐留

RAPO 涉及亞太多國的數據，可能有數據駐留要求：
- Claude API 的數據處理位置：Anthropic 的數據中心在哪裡？是否符合各國數據保護法？
- Azure OpenAI 的數據處理：可以選擇 Azure 區域，但需要確認符合各國要求
- LDAP 查詢數據：跨國 AD 查詢可能涉及跨境數據傳輸
- **建議**：在 Phase A 的安全架構文件中加入數據流向圖，標明數據處理位置；諮詢 Legal 團隊確認合規性

#### 11.1.5 Disaster Recovery 和 Business Continuity

原報告未討論 IPA Platform 本身的 DR/BC 策略：
- 如果 IPA Platform 宕機，依賴它的自動化流程如何 fallback？
- RabbitMQ 空殼意味著沒有消息持久化，服務中斷期間的事件會丟失
- **建議**：所有通過 IPA 自動化的流程必須保留手動 fallback 路徑；在 Phase C 中加入 DR 策略

#### 11.1.6 LLM 模型更新和 Prompt Drift

AI 模型會定期更新，可能導致行為變化（Prompt Drift）：
- Claude 模型更新後，Agent 的行為可能不同
- PatternMatcher 的規則可能需要隨模型更新調整
- **建議**：建立 Prompt 版本控制和回歸測試機制；在模型更新時先在測試環境驗證

#### 11.1.7 內部政治與利益相關者管理

引入 AI Agent 平台涉及多個利益相關者：
- IT Operations 團隊：直接使用者，但可能擔心被取代
- IT Security 團隊：審批權限的守門人
- ServiceNow Admin：需要配合 Webhook 設置
- IT Director：預算審批者
- 其他部門（如 HR）：可能對 AD 帳號自動化有意見（如入離職流程）
- **建議**：建立利益相關者矩陣，針對每個群體制定溝通策略
