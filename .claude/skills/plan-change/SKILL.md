# Plan Change - 功能變更規劃

為現有功能的修改或增強建立符合項目規範的 CHANGE 規劃文件。

## 執行流程

### Step 1: 掃描現有編號

```
1. Glob: claudedocs/4-changes/feature-changes/CHANGE-*.md
   → 找到最新編號，自動遞增
2. Read: 最近 1 個 CHANGE 文件作為格式參考
```

告知用戶下一個可用編號（例：CHANGE-031）。

### Step 2: 收集變更資訊

詢問用戶以下資訊：

1. **變更標題**（中文，簡短描述變更內容）
2. **變更類型**（UI Enhancement / Feature / Refactor / Performance / Security）
3. **優先級**（High / Medium / Low）
4. **影響範圍**（受影響的頁面/組件/服務）
5. **變更背景**（為什麼需要這個變更？）
6. **變更內容概要**（要改什麼？）

### Step 3: 自動分析影響範圍

根據用戶描述的影響範圍，掃描相關代碼：

```
1. Grep/Glob: 找到受影響的文件
2. 識別是否涉及：
   - i18n 翻譯文件（messages/）
   - Prisma Schema（prisma/schema.prisma）
   - API 路由（src/app/api/）
   - 組件（src/components/）
   - 服務層（src/services/）
```

### Step 4: 生成 CHANGE 文件

**路徑**: `claudedocs/4-changes/feature-changes/CHANGE-{NNN}-{kebab-case-title}.md`

**必須遵循的格式**：

```markdown
# CHANGE-{NNN}: {變更標題}

> **日期**: {YYYY-MM-DD}
> **狀態**: ⏳ 待實作
> **優先級**: {High/Medium/Low}
> **類型**: {變更類型}
> **影響範圍**: {影響範圍摘要}

---

## 變更背景

{背景說明：為什麼需要這個變更}

## 變更內容

### {變更項目 1}
{具體描述}

### {變更項目 2}（如有多個）
{具體描述}

## 技術設計

### 修改範圍

| 文件 | 變更內容 |
|------|----------|
| `{路徑}` | {修改說明} |

| 語言 | 文件 | 需要更新的 Key |
|------|------|---------------|
| en | `messages/en/{file}.json` | {key 清單} |
| zh-TW | `messages/zh-TW/{file}.json` | {同上} |

### 資料庫影響（如適用）

{Prisma Schema 變更說明}

## 設計決策

## 影響範圍評估

### 文件影響清單

| 文件路徑 | 類型 | 說明 |
|----------|------|------|
| `{路徑}` | 🆕 新增 / 🔧 修改 / ❌ 刪除 | {說明} |

- {分析說明}

## 驗收標準

| # | 驗收項目 | 驗收標準 | 優先級 |
|---|----------|----------|--------|

```

向用戶展示生成結果，等待確認後寫入。

### Step 5: 輸出摘要

```markdown
## ✅ 變更規劃完成

| 項目 | 內容 |
|------|------|
| 文件 | `claudedocs/4-changes/feature-changes/CHANGE-{NNN}-{name}.md` |
| 編號 | CHANGE-{NNN} |
| 狀態 | ⏳ 待實作 |

### 下一步
- 開始實作此變更
- 實作完成後更新狀態為 ✅ 已完成
```

## 文件格式變體

### 簡單變更（影響 < 5 個文件）

可以省略以下章節：
- 影響範圍評估
- 測試場景

### 複雜變更（影響 > 10 個文件或跨多模組）

應額外包含：
- 實施計劃（分階段）
- 風險評估
- 回滾計劃

## 重要規則

- 所有文件內容使用**繁體中文**
- 編號必須連續遞增，不可跳號
- 文件名使用 kebab-case 英文
- 每個步驟的輸出都必須等用戶確認後才寫入
- 掃描到的影響文件必須真實存在於代碼庫中
