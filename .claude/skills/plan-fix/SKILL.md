# Plan Fix - Bug 修復規劃

為 Bug 修復建立符合項目規範的 FIX 規劃文件。

## 執行流程

### Step 1: 掃描現有編號

```
1. Glob: claudedocs/4-changes/bug-fixes/FIX-*.md
   → 找到最新編號，自動遞增
2. Read: 最近 1 個 FIX 文件作為格式參考
```

告知用戶下一個可用編號（例：FIX-036）。

### Step 2: 收集 Bug 資訊

詢問用戶以下資訊：

1. **Bug 簡述**（中文，簡短描述問題）
2. **發現方式**（測試 / 用戶回報 / 代碼審查 / Playwright E2E）
3. **影響頁面/功能**（具體的 URL 路徑或功能名稱）
4. **優先級**（高 / 中 / 低）
5. **問題描述**（詳細描述看到了什麼）
6. **重現步驟**（如果知道的話）

### Step 3: 自動分析相關代碼

根據用戶描述的影響範圍，掃描相關代碼：

```
1. Grep/Glob: 根據影響頁面找到相關文件
2. 識別可能的根本原因區域：
   - 組件代碼（src/components/）
   - API 路由（src/app/api/）
   - 服務層（src/services/）
   - i18n 翻譯（messages/）
   - Prisma Schema / 查詢
```

### Step 4: 生成 FIX 文件

**路徑**: `claudedocs/4-changes/bug-fixes/FIX-{NNN}-{kebab-case-description}.md`

**必須遵循的格式**：

```markdown
# FIX-{NNN}: {Bug 簡述}

> **建立日期**: {YYYY-MM-DD}
> **發現方式**: {測試/用戶回報/代碼審查/Playwright E2E}
> **影響頁面/功能**: {範圍}
> **優先級**: {高/中/低}
> **狀態**: 🚧 待修復

---

## 問題描述

{詳細問題說明}

{如有多個子問題，使用表格}：

| # | 問題 | 嚴重度 | 影響頁面 |
|---|------|--------|----------|
| BUG-1 | {描述} | {高/中/低} | {頁面} |

---

## 重現步驟

1. {步驟 1}
2. {步驟 2}
3. 觀察現象：{bug 表現}

---

## 根本原因

{根因分析}

### {子原因 1}（如有多個原因）
- {具體分析}

---

## 解決方案

### {Bug 1} 修復：{方案名稱}

{修復說明，可含代碼範例}

---

## 修改的檔案

| 檔案 | 修改內容 |
|------|----------|
| `{路徑}` | {修改說明} |

---

## 測試驗證

修復完成後需驗證：

- [ ] {驗證項目 1}
- [ ] {驗證項目 2}

---

*文件建立日期: {YYYY-MM-DD}*
*最後更新: {YYYY-MM-DD}*
```

向用戶展示生成結果，等待確認後寫入。

### Step 5: 輸出摘要

```markdown
## ✅ Bug 修復規劃完成

| 項目 | 內容 |
|------|------|
| 文件 | `claudedocs/4-changes/bug-fixes/FIX-{NNN}-{name}.md` |
| 編號 | FIX-{NNN} |
| 狀態 | 🚧 待修復 |

### 下一步
- 開始修復此 Bug
- 修復完成後更新狀態為 ✅ 已修復
- 更新「修改的檔案」和「解決方案」章節
```

## 文件格式變體

### 單一 Bug（簡單）

省略子問題表格，直接描述問題。

### 多個相關 Bug（複合）

使用子問題表格，每個 Bug 獨立描述根因和方案，並加入：
- 修復優先級表格
- 修復順序建議

## 重要規則

- 所有文件內容使用**繁體中文**
- 編號必須連續遞增，不可跳號
- 文件名使用 kebab-case 英文
- 每個步驟的輸出都必須等用戶確認後才寫入
- 「根本原因」和「解決方案」在建立時可標記為「待分析」，修復過程中再補充
- 「修改的檔案」在建立時可根據分析預填，修復後必須更新為實際修改的文件
